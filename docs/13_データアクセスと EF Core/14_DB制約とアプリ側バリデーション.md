# DB制約とアプリ側バリデーション

アプリ側バリデーションは、ユーザーに分かりやすくエラーを返すために使います。DB制約は、最後にデータの整合性を守るために使います。

| 観点 | アプリ側 | DB側 |
| --- | --- | --- |
| 必須 | DTO で入力チェック | `NOT NULL` |
| 文字数 | `StringLength` など | カラム長 |
| 重複 | 事前確認 | 一意制約 |
| 関連 | ID の存在確認 | 外部キー制約 |

アプリ側でチェックしていても、同時リクエストでは重複登録が起きることがあります。そのため、メールアドレスなどの重複禁止は一意制約でも守ります。

DB制約に違反した場合は、内容に応じて `400 Bad Request` や `409 Conflict` に変換します。例外メッセージをそのまま返してはいけません。

## よくある失敗

- DTO でチェックしているから DB制約は不要だと思う
- 一意制約がなく、同時登録で重複データが入る
- 外部キー制約を避けた結果、存在しない親データを参照する
