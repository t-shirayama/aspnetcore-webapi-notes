# 日付と金額と ID の型設計

DTO の型は、API を使う側との契約になります。C# で扱いやすいだけでなく、意味が伝わる型を選びます。

| 値 | よく使う型 | 考え方 |
| --- | --- | --- |
| 日付だけ | `DateOnly` | 誕生日、締切日、営業日 |
| 日時 | `DateTimeOffset` | 作成日時、更新日時、期限日時 |
| 金額 | `decimal` | 誤差を避けたい数値 |
| 内部 ID | `int` / `long` | DB の主キーとして扱いやすい |
| 外部公開 ID | `Guid` / 文字列 ID | 推測されにくさや連携を重視 |

`DateTime` はタイムゾーンの意味が曖昧になりやすいため、API の入出力では `DateTimeOffset` を使うと意図を表しやすくなります。

金額に `double` や `float` を使うと、小数の誤差が問題になります。金額や単価は `decimal` を基本にします。

ID は、内部の DB 主キーをそのまま公開してよいかを考えます。公開 API や外部連携では、連番 ID より `Guid` や別の公開 ID を使うことがあります。

## よくある失敗

- 日付だけでよい項目に日時を使い、タイムゾーンでずれる
- 金額を `double` で扱い、丸め誤差が出る
- 外部に見せたくない内部 ID をレスポンスに含める
