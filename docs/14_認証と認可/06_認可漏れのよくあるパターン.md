# 認可漏れのよくあるパターン

認可漏れは、API で特に危険な不具合です。

起きやすいパターン:

- 一覧 API では絞っているが、詳細 API では所有者チェックを忘れる
- 更新 API だけ `[Authorize]` を付け忘れる
- 管理者用 API のルートを知っていれば呼べてしまう
- クライアント側のボタン非表示だけで制御している
- ID を変えると他人のデータを見られる
- 作成時は確認するが、更新・削除時の確認を忘れる
- `[Authorize]` だけで、管理者ロールや所有者確認をしていない

認可は UI ではなく **API 側で必ず確認する** 必要があります。

特に、対象データがログインユーザーのものかを確認する処理は忘れやすいです。

## 所有者チェック

`[Authorize]` は「ログインしているか」を確認しますが、「その記事を見てよいか」までは自動では判断しません。

```csharp
if (article.OwnerUserId != currentUserId)
{
    return Results.Forbid();
}
```

`GET /articles/{id}`、`PUT /articles/{id}`、`DELETE /articles/{id}` のように ID で対象を指定する API では、対象データの所有者とログインユーザーを比較します。

管理者だけが使う API では Role や Policy が向いています。自分のデータだけ操作できる API では、所有者チェックが必要です。
