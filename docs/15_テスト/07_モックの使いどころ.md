# モックの使いどころ

モックは、テスト対象の外側にある依存を置き換えるために使います。

モックを使う目的は、テストを速く、安定させ、確認したいロジックに集中することです。

## モックに向いているもの

モックに向いているのは、テスト中に本物を使うと遅い、不安定、危険、準備が重いものです。

- メール送信
- 外部 API 呼び出し
- 現在時刻
- ファイル操作
- 認証ユーザー情報
- メッセージキュー
- 決済など副作用が大きい処理

例えば、メール送信 Service の本物を呼ぶと、テストのたびにメールが送られてしまいます。この場合は、メール送信インターフェイスをモックに差し替えます。

## DB はモックしすぎない

DB 周りは、モックしすぎると実際の挙動から離れます。

EF Core の LINQ、制約、トランザクション、SQL 変換は、モックでは再現しづらいです。

そのため、Repository や DbContext 周りは、SQLite や PostgreSQL などのテスト DB を使った結合テストの方が有効なことがあります。

## モックの使いすぎに注意する

モックが多すぎるテストは、実装の詳細に強く依存します。

例えば「このメソッドが 1 回呼ばれたか」ばかり確認すると、リファクタリングで壊れやすくなります。

できるだけ、呼び出し回数よりも **最終的な結果や状態** を確認します。

モックは便利ですが、目的は「本物に近い保証」ではなく「狭い範囲の判断を安定して確認すること」です。
