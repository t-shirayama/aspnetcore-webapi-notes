# 入力値検証と業務ルール検証

入力値検証と業務ルール検証は分けて考えます。

入力値検証は、リクエストの形や単純な値の妥当性を確認します。

- 必須チェック
- 文字数チェック
- 数値範囲チェック
- 日付形式のチェック

業務ルール検証は、システムのルールとして許可できるかを確認します。

- メールアドレスが重複していないか
- 開始日が終了日より前か
- 注文をキャンセルできる状態か
- ログインユーザーが対象データを操作できるか

入力値検証は DTO の近く、業務ルール検証は Service やドメインロジックの近くに置くと整理しやすいです。

## どこに書くか

入力値検証は、`DataAnnotations` や FluentValidation で Request DTO の近くに置くと分かりやすいです。

```csharp
public record CreateArticleRequest(
    [Required]
    [MaxLength(100)]
    string Title,
    string Body
);
```

業務ルール検証は、DB や現在の状態を見ないと判断できないことが多いため、Service に置くことが多いです。

```csharp
if (await repository.ExistsByTitleAsync(request.Title))
{
    return CreateArticleResult.Conflict("同じタイトルの記事が既に存在します。");
}
```

## レスポンスの考え方

入力値検証の失敗は、クライアントがリクエストを直せば解決できるため、`400 Bad Request` や `422 Unprocessable Content` が候補になります。

業務ルール検証の失敗は、状況によってステータスコードを分けます。

| 状況 | 候補 |
| --- | --- |
| 対象が存在しない | `404 Not Found` |
| 重複している | `409 Conflict` |
| 権限がない | `403 Forbidden` |
| 状態的に処理できない | `409 Conflict` または `422 Unprocessable Content` |

大切なのは、エラーの種類ごとにクライアントが次の判断をできる形にすることです。
