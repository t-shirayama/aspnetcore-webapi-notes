# テストでよく見る観点

Web API のテストでは、正常系だけでなく異常系も確認します。

## 正常系

正常系では、期待した操作が成功することを確認します。

- 一覧を取得できる
- 1 件を取得できる
- 作成できる
- 更新できる
- 削除できる
- レスポンス DTO の形が正しい

作成 API では `201 Created`、削除 API では `204 No Content` など、ステータスコードも確認します。

## 入力エラー

入力エラーでは、クライアントが修正できる問題を確認します。

- 必須項目がない
- 文字数が長すぎる
- 数値範囲を超えている
- 日付の前後関係が不正
- JSON の形式が不正

この場合は `400 Bad Request` や `422 Unprocessable Content` を返す設計が候補になります。

## 業務エラー

業務エラーでは、形式は正しいが業務ルール上処理できないケースを確認します。

- 重複登録
- 状態不一致
- 在庫不足
- すでに削除済み
- 他人のデータを操作しようとしている

重複や競合は `409 Conflict`、対象がない場合は `404 Not Found` など、設計したステータスコードに合っているかを見ます。

## 認証と認可

認証・認可では、誰が呼んだときに何が許可されるかを確認します。

- 未ログインなら `401`
- ログイン済みだが権限不足なら `403`
- 自分のデータは操作できる
- 他人のデータは操作できない
- 管理者だけが実行できる

認可漏れは重大な不具合になりやすいため、統合テストで確認する価値が高いです。

## 例外処理

例外処理では、未処理例外が外部に漏れないことを確認します。

- スタックトレースを返していない
- Problem Details の形式になっている
- ログには調査に必要な情報が残る
- レスポンスには秘密情報が含まれない

エラー時のテストは、API の品質を大きく左右します。
