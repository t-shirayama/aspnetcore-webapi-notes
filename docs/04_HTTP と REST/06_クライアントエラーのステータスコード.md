# クライアントエラーのステータスコード

クライアントエラーは、リクエスト側が修正できる問題を表します。

| コード | 使う場面 |
| --- | --- |
| `400 Bad Request` | リクエスト形式や基本的な入力値が不正 |
| `401 Unauthorized` | 認証されていない |
| `403 Forbidden` | 認証済みだが権限がない |
| `404 Not Found` | 対象リソースが存在しない |
| `409 Conflict` | 重複や競合がある |
| `422 Unprocessable Content` | 形式は正しいが意味的に処理できない |

実務では、入力エラーをすべて `400` に寄せる設計もあります。一方で、JSON の形式は正しいが業務的に受け付けられない場合に `422` を使う設計もあります。

大切なのは、**プロジェクト内で使い分けを統一すること** です。

## `409 Conflict`

`409 Conflict` は、リクエスト自体の形は正しいものの、現在のサーバー側の状態と衝突して処理できない場合に使います。

よくある例:

- 既に使われているメールアドレスでユーザー登録しようとした
- 楽観ロックのバージョンが古いまま更新しようとした
- 注文がキャンセル済みなのに発送しようとした

入力値の書式ミスではなく、**状態の競合** を伝えたいときに候補になります。

## `422 Unprocessable Content`

`422 Unprocessable Content` は、リクエストの形式は読めるが、意味的に受け付けられない場合に使うことがあります。

```text
400: JSON が壊れている、型が違う、必須項目がない
422: 開始日が終了日より後、業務上選べない組み合わせ
```

ただし、ASP.NET Core の API では入力エラーを `400 Bad Request` にまとめる設計も自然です。`422` を使うなら、OpenAPI とエラー形式にも明記して、クライアントが迷わないようにします。
