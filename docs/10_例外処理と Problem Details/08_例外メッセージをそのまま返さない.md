# 例外メッセージをそのまま返さない

例外メッセージやスタックトレースを、そのまま API レスポンスに返してはいけません。

例外には、SQL、ファイルパス、接続文字列、内部クラス名、外部サービスの応答など、外部に見せるべきでない情報が含まれることがあります。

よくある失敗は、すべての例外を `try-catch` で握りつぶして、例外メッセージをそのまま返すことです。

```csharp
catch (Exception ex)
{
    return BadRequest(ex.Message);
}
```

この書き方では、クライアントに内部事情を漏らすうえ、`400 Bad Request` と `500 Internal Server Error` の区別も壊れます。

基本方針は次の通りです。

| 情報 | 扱い |
| --- | --- |
| 例外の詳細 | ログに残す |
| スタックトレース | ログに残す |
| 入力エラー | ProblemDetails として返す |
| 業務エラー | 状況に応じて `400` / `404` / `409` などで返す |
| システムエラー | 汎用的な `500` として返す |

レスポンスには、利用者が次に何をすればよいか分かる情報だけを返します。調査に必要な詳細はログへ残します。
